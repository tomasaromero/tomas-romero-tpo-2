Desarrollo Web

Construyendo interfaces accesibles

22 de Octubre, 2025

En el desarrollo web, a menudo nos centramos en la estética, la velocidad de carga o las últimas tendencias de frameworks. Sin embargo, hay un pilar fundamental que a veces se pasa por alto: la accesibilidad (a11y). Construir interfaces accesibles no es una característica adicional; es un requisito fundamental para garantizar que la web sea un espacio inclusivo para todos, independientemente de sus capacidades físicas o cognitivas.

1. HTML Semántico: La base de todo

Antes de cualquier framework de CSS o JavaScript, la accesibilidad comienza con un HTML sólido y semántico. Los lectores de pantalla y otros dispositivos de asistencia dependen de la estructura del DOM para entender la página.

No uses <div> para todo: Si algo funciona como un botón, usa la etiqueta <button>. Si es un enlace, usa <a>. Para la navegación, usa <nav>, y para el contenido principal, <main>.

Jerarquía de encabezados: Usa <h1> (solo uno por página), <h2>, <h3>, etc., en un orden lógico. No uses encabezados solo para cambiar el tamaño de la fuente.

2. Navegación por Teclado

No todos los usuarios utilizan un mouse. Las personas con discapacidades motoras dependen exclusivamente del teclado para navegar.

Visibilidad del foco: Asegúrate de que el estado :focus sea claramente visible. Muchos resets de CSS eliminan el outline por defecto; si lo haces, debes proporcionar una alternativa visual clara.

Orden lógico: El orden en que se navega con la tecla Tab debe seguir el flujo visual de la página (de izquierda a derecha, de arriba abajo).

3. Contraste y Color

El diseño visual debe ser perceptible para personas con baja visión o daltonismo.

Contraste mínimo: Las directrices WCAG exigen una relación de contraste mínima (generalmente 4.5:1 para texto normal). Herramientas como Lighthouse en Chrome DevTools pueden auditar esto automáticamente.

El color no debe ser el único indicador: No uses solo el color rojo para indicar un error. Acompáñalo de un icono, un texto descriptivo o un borde grueso.

4. ARIA para cuando el HTML no es suficiente

ARIA (Accessible Rich Internet Applications) proporciona atributos para mejorar la accesibilidad de componentes dinámicos (como pestañas, acordeones o modales) que el HTML estándar no cubre.

aria-label: Da un nombre accesible a un elemento que no tiene texto visible (ej. un botón de icono aria-label="Cerrar").

role: Define el propósito del elemento (ej. role="alert" para mensajes de error dinámicos).

Conclusión

La accesibilidad web no es un favor que hacemos, es una responsabilidad profesional. Integrar estas prácticas desde el inicio del proyecto no solo abre nuestro sitio a millones de nuevos usuarios, sino que también mejora el SEO y, francamente, nos convierte en mejores desarrolladores.
